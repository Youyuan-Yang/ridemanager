# RideManager - 一个综合检测上位机

## 介绍

RideManager 是一个检测上位机，它目标部署在rk3588平台上，
提供电动车的出行安全检测功能。它集成了多种传感器和算法，能够实时监测电动车的状态和环境。

## 外部设备&设计

1. 前向摄像头**CAM_FRONT**：专注于前方目标检测、距离代理分析和时间窗口趋势分析；车道线检测功能已放弃，不再作为正式处理链路。
2. 雷达传感器**RADAR**：用于测量用户的心率和呼吸，使用蓝牙通讯下位机。
3. 面部摄像头**CAM_FACE**：用于监测用户的面部表情和疲劳状态，使用深度学习模型来分析面部特征。
4. 后部鱼眼摄像头**CAM_BACK**：用于监测后方的交通状况，提供广视角，使用深度学习模型来识别和分析图像数据。
5. 刹车驱动器**BRAKE**：用于控制电动车的刹车系统，根据检测到的危险情况自动触发刹车。
6. 语音播报系统**SPEAKER**：用于向用户提供实时的安全提示和警告，使用预录好的语音提示。
7. 陀螺仪下位机**GYRO**：用于监测电动车的倾斜和转向，提供稳定性分析和安全建议。

## 处理链路

- 每个模块都单独设计，不紧耦合
- 每个摄像头(CAM_FRONT, CAM_FACE, CAM_BACK)都分别进行：
    1. 图像采集(使用高性能丢帧不缓存机制)
    2. 图像预处理(OpenCV)
    3. 深度学习推理（onnx 或者 rknn）
    4. 汇总数据到主控模块
    注意：每个摄像头的算法与模型都可以独立，不互相依赖，主控模块只负责数据汇总和决策。
- 其他传感器（RADAR, GYRO）通过蓝牙或串口通信将数据发送到主控模块。
- 主控模块根据所有传感器的数据进行综合分析，判断是否存在安全风险，并控制刹车驱动器和语音播报系统进行相应的操作。
- 状态、数据与决策：全部写入psql里，供前端调用
- 手机 App 通过蓝牙自动连接 RideManager，默认同步最近 24 小时数据库数据，支持分页加载更多历史数据和设置变更请求。
- 正式运行时，单个摄像头打开失败只输出 warning，并禁用该摄像头对应处理链路；其他摄像头和传感器继续运行。

## 项目结构

src/ - 项目源代码
- core/ - 主控模块代码
- camera/ - 摄像头处理模块代码
- models/ - onnx和rknn统一抽象推理模块代码
- sensors/ - 其他传感器处理模块代码，每个传感器单独封装通讯和数据处理逻辑
- actuators/ - 刹车驱动器、语音播报等执行器占位与控制接口
- utils/ - 工具类和公共函数代码
- data/ - 数据处理和存储相关代码
- appsync/ - 手机 App 蓝牙同步协议、数据库分页读取和设置变更接入

config.toml - 项目配置文件，包含摄像头参数、模型路径、传感器配置等信息

cpp_rknn/ - RKNN C++桥接层代码，提供在RK3588上使用RKNN的接口，使用CMake构建
    推理传递0拷贝，对接c# native memory

## **注意事项**

- 模块之间的接口设计要清晰，确保模块独立性和可替换性。
- 图像处理注意0拷贝和高性能，避免不必要的内存复制。
- 单个文件不能太长，需要适当分文件。
- 代码风格要统一，每个函数或者类前都需要有简短的注释。
- 不要手写ORM代码，使用ef core自动生成数据库访问层。
- 不要过度设计，不要过度分层和分文件夹
- **不要新建太多项目** 只分主项目和测试项目
- 代码追求纯c#，onnxruntime和opencv也使用c#版本的库，尽量避免python脚本和其他语言的代码，保持项目的纯净和一致性。
